# Snap Services and Daemon Debugging

## List snap services
```bash
snap services                  # all snap services
snap services <name>           # services for a specific snap
```

Output columns: Service, Startup (enabled/disabled), Current (active/inactive), Notes.

## Control snap services
```bash
snap start <name>              # start all services of a snap
snap start <name>.<service>    # start a specific service
snap stop <name>               # stop all services
snap stop <name>.<service>     # stop a specific service
snap restart <name>            # restart all services
snap restart <name>.<service>  # restart a specific service
```

## Enable and disable
```bash
snap start --enable <name>.<service>   # enable and start
snap stop --disable <name>.<service>   # disable and stop
```

## Underlying systemd units
Snap services are managed via systemd. The unit files are generated by snapd.

```bash
# View the systemd unit
systemctl cat snap.<name>.<service>.service

# Check unit status
systemctl status snap.<name>.<service>.service

# View recent logs for the service
journalctl -u snap.<name>.<service>.service --since "10 min ago"
journalctl -u snap.<name>.<service>.service -f  # follow logs

# List all snap-related systemd units
systemctl list-units 'snap.*'
systemctl list-units 'snap.*' --state=failed  # only failed units
```

## Timer services
Snaps can define timer-based services (like cron jobs).

```bash
# List snap timers
systemctl list-timers 'snap.*'

# Check a specific timer
systemctl status snap.<name>.<service>.timer
```

## Socket-activated services
```bash
systemctl list-sockets 'snap.*'
systemctl status snap.<name>.<service>.socket
```

## Debugging service failures
1. Check `snap services <name>` to see if the service is active.
2. Check `systemctl status snap.<name>.<service>.service` for exit code and recent output.
3. Check `journalctl -u snap.<name>.<service>.service -n 50` for logs.
4. Check if the service was disabled: `snap services <name>` shows Startup column.
5. Try restarting: `snap restart <name>.<service>`.
6. If service keeps crashing, check for AppArmor denials: `dmesg | grep DENIED`.
7. Check resource limits: `systemctl show snap.<name>.<service>.service | grep -i limit`.

## Snap daemon types
- **simple**: default, process runs in foreground.
- **forking**: process forks and parent exits.
- **oneshot**: runs once and exits.
- **notify**: process signals readiness via sd_notify.

## Common issues
- **Service not starting after install**: Some services need interface connections before they can start.
- **Service restarts in a loop**: Check logs for crash reason. May be a missing dependency or permission issue.
- **Service disabled after refresh**: Check if `snap stop --disable` was called, or if the snap.yaml changed the daemon definition.
- **Watchdog killing service**: Check `WatchdogTimestamp` in `systemctl show` output.
